Full Analysis & Change Plan: Adding inAppUpdatePriority Support
Background: What's Missing
The library currently forces you to hardcode updateType: IAUUpdateKind.FLEXIBLE or IAUUpdateKind.IMMEDIATE in JS before calling startUpdate(). The Google Play Developer API's inAppUpdatePriority field (0–5, set per-release via the Play API) is already sent inside AppUpdateInfo at the native level — the library just never reads it, never passes it to JS, and never uses it to decide update type automatically. Here's every change needed, layer by layer.

1. android/.../SpReactNativeInAppUpdatesModule.java — The Core Fix
This is the most important file. Right now checkNeedsUpdate() resolves with a map that's missing updatePriority, clientVersionStalenessDays, and isImmediateUpdateAllowed/isFlexibleUpdateAllowed. startUpdate() accepts an updateType int but ignores priority entirely.
Changes needed:
a) In checkNeedsUpdate() — enrich the resolved map:
java// ADD these fields to the WritableMap you resolve with:
map.putInt("updatePriority", appUpdateInfo.updatePriority());
map.putInt("clientVersionStalenessDays",
    appUpdateInfo.clientVersionStalenessDays() != null
      ? appUpdateInfo.clientVersionStalenessDays() : -1);
map.putBoolean("isImmediateUpdateAllowed",
    appUpdateInfo.isUpdateTypeAllowed(AppUpdateType.IMMEDIATE));
map.putBoolean("isFlexibleUpdateAllowed",
    appUpdateInfo.isUpdateTypeAllowed(AppUpdateType.FLEXIBLE));
b) In startUpdate() — add auto-resolution by priority:
Accept an optional updateType int from JS but also accept a special sentinel value (e.g. -1 = "auto-decide from priority"). When auto mode is set, read appUpdateInfo.updatePriority() and apply a mapping like:
java// Priority-based auto selection (add as a helper method):
private int resolveUpdateType(AppUpdateInfo info, int requestedType) {
    if (requestedType != UPDATE_TYPE_AUTO) return requestedType;
    int priority = info.updatePriority();
    if (priority >= 4) return AppUpdateType.IMMEDIATE;
    if (priority >= 2) return AppUpdateType.FLEXIBLE;
    return AppUpdateType.FLEXIBLE; // default for 0-1
}
c) Add the AppUpdateOptions.setAllowAssetPackDeletion() support (new Google Play API feature, currently unused):
javaAppUpdateOptions options = AppUpdateOptions
    .newBuilder(resolvedUpdateType)
    .setAllowAssetPackDeletion(allowAssetPackDeletion) // from JS params
    .build();
appUpdateManager.startUpdateFlowForResult(appUpdateInfo, activity, options, ...);
The library currently uses the deprecated startUpdateFlowForResult(appUpdateInfo, updateType, activity, requestCode) signature. It needs to be migrated to the AppUpdateOptions builder pattern (current API as of com.google.android.play:app-update:2.1.0).
d) Bump the Gradle dependency in android/build.gradle:
groovy// Change from whatever older version to:
implementation 'com.google.android.play:app-update:2.1.0'

3. src/InAppUpdates.android.ts — Priority-Aware Logic Layer
This is the JS-side Android implementation class. Currently checkNeedsUpdate() receives the native result and returns it mostly as-is, and startUpdate() passes updateType straight through to native.
Changes needed:
a) In checkNeedsUpdate() result handling — pass through the new updatePriority and clientVersionStalenessDays fields in the returned object (they'll now come from native, so it's mostly just not stripping them out).
b) Add a convenience method startUpdateWithPriority() (or modify startUpdate() to support auto mode):
typescript// If updateType is not provided (or is IAU_UPDATE_TYPE_AUTO),
// pass -1 to native to let it auto-select based on priority
startUpdate(updateOptions: StartUpdateOptions = {}) {
  const { updateType = IAU_UPDATE_TYPE_AUTO, allowAssetPackDeletion = false } = updateOptions;
  return this.nativeModule.startUpdate(updateType, allowAssetPackDeletion);
}
c) Add getPriority() helper — a convenience method that calls checkNeedsUpdate() and returns just updatePriority so consumers can make their own decisions:
typescriptasync getPriority(): Promise<number> {
  const result = await this.checkNeedsUpdate();
  return (result as AndroidNeedsUpdateResponse).updatePriority ?? 0;
}

4. src/InAppUpdatesBase.ts — Base Class / Abstract Definitions
Changes needed:
Add allowAssetPackDeletion to the shared AndroidStartUpdateOptions base type, and document that updateType is now optional (defaults to auto-priority-based). The base class enforces the interface contract that both Android and (stub) iOS sides must satisfy.

This is the public API surface used by consumers.

typescript// ADD to exports:

Without this update, consumers using TypeScript won't know that updatePriority is available on the result, and they'll get type errors if they try to pass allowAssetPackDeletion.

6. src/index.js — JS Entry Point / Re-exports

7. src/utils.ts — Add Priority Mapping Utility
Add a pure utility function that maps a raw updatePriority int to an IAUUpdateKind so consumers who want to do the decision themselves in JS (rather than delegating to native) can do so consistently:
typescriptexport function updateKindFromPriority(
  priority: number,
  stalenessDays?: number
): IAUUpdateKind {
  if (priority >= 4) return IAUUpdateKind.IMMEDIATE;
  if (priority >= 2) return IAUUpdateKind.FLEXIBLE;
  // Optional: staleness fallback — if low priority but very stale, force immediate
  if (stalenessDays !== undefined && stalenessDays >= 14) return IAUUpdateKind.FLEXIBLE;
  return IAUUpdateKind.FLEXIBLE;
}
This gives you full control in your own app code to override the thresholds.

How Your App Code Looks After These Changes
Option A — Full auto (let native decide based on Play priority):
typescriptconst result = await inAppUpdates.checkNeedsUpdate();
if (result.shouldUpdate) {
  // No updateType specified — native reads updatePriority() and decides
  await inAppUpdates.startUpdate({});
}
Option B — JS-side decision using the utility:
typescriptimport { updateKindFromPriority } from 'sp-react-native-in-app-updates/utils';

const result = await inAppUpdates.checkNeedsUpdate();
if (result.shouldUpdate && Platform.OS === 'android') {
  const { updatePriority, clientVersionStalenessDays } = result;
  const updateType = updateKindFromPriority(updatePriority, clientVersionStalenessDays);
  await inAppUpdates.startUpdate({ updateType });
}
